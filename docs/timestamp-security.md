## 1. 先搞懂：时间戳到底在干什么

### 一句话理解

> 时间戳服务器（TSA）证明的是：**在某个时刻，某个哈希值对应的数据已经存在了。**

它不关心你的数据是什么，也不验证你的身份。它只做一件事：

```
你提交：hash(file) + "请证明这一刻它已存在"
TSA 返回：[hash(file) + 当前时间] + TSA 的数字签名
```

### 时间戳和 NTP 的本质区别

| | 时间戳服务器 (TSA) | NTP |
|---|---|---|
| 目的 | 证明数据在某个时间点前已存在 | 同步计算机系统时钟 |
| 输出 | 有数字签名的令牌（可作为证据） | 一个时间数值（仅用于校准） |
| 可向第三方验证 | ✅ 任何人可验证签名 | ❌ 只能信来源，无法举证 |
| 标准 | RFC 3161 | RFC 5905 |

> 详细对比见 [时间戳服务器与 NTP 的区别](./timestamp-vs-ntp.md)

## 2. 常见疑虑一：哈希碰撞导致伪造

### 疑虑

> "两个不同的文件可能算出相同的 SHA-256 值，我可以找一个碰撞文件来匹配别人的时间戳令牌。"

### 解答

SHA-256 的设计目标之一就是**抗碰撞性（Collision Resistance）**：

| 攻击类型 | 含义 | 计算量 | 可行性 |
|---|---|---|---|
| 任意碰撞 | 找到任意两个文件 X ≠ Y，使 hash(X) = hash(Y) | 2¹²⁸ | 理论上可能，但当前算力不可行 |
| 第二原像攻击 | 给定文件 A，找到文件 B ≠ A 使 hash(B) = hash(A) | 2²⁵⁶ | **当前算力绝对不可行** |

时间戳令牌封存的是具体文件的哈希值。你要伪造的场景属于**第二原像攻击**——找到一个不同的文件恰好产生相同的哈希。这在 SHA-256 下需要 2²⁵⁶ 次尝试，以当前所有比特币矿机的总算力计算，也需要**远超宇宙年龄的时间**。

### 万一 SHA-256 以后被攻破了呢？

实践中已有预案：

- **多哈希并用**：同时用 SHA-256 + SHA-384 + SM3 打时间戳，攻破一个还有另一个
- **时间戳续期**：旧算法弱化时，对原始文件用新哈希重新打时间戳
- **RFC 3161 支持算法协商**：可随时切换到更强的哈希算法

## 3. 常见疑虑二：本地篡改哈希欺骗 TSA

### 疑虑

> "哈希在本地计算，我 hook 掉 SHA-256 函数，让它对任何文件都返回同一个已知哈希值，然后骗过 TSA 拿到有效时间戳。"

### 解答——这是核心问题，也是最大的误解

**这个攻击能做到什么？**

```
受害者文件 A 的哈希：h(A) = 0xABC...
攻击者文件 B 的哈希：h(B) = 0xDEF...

攻击者 hook 本地 SHA-256：
  输入文件 B → hook 返回 0xABC... → 发送给 TSA
  TSA 返回：[0xABC... + 时间] + TSA 签名 → 有效令牌 ✅
```

攻击者确实能拿到一张**语法上有效的时间戳令牌**。

**但它通不过验证——而且是必然失败。**

```
验证者（在你的机器之外，独立运行）：
  步骤 1：拿到你的文件 B + 你的时间戳令牌
  步骤 2：在自己的系统上计算 SHA-256(B) → 得到 0xDEF...（注意，这里没有 hook）
  步骤 3：解封时间戳令牌，读出里面封存的哈希 → 0xABC...
  步骤 4：比较 0xDEF... ≠ 0xABC... → ❌ 验证失败
```

**关键点：验证时用的是验证者自己算的哈希，不是你的声明。** 你的 hook 只影响你自己的机器，影响不了验证者的计算。

### 一张图看懂

```
你的机器（不可信）                    TSA（可信）                 验证者（可信）

正常流程：
文件 → SHA-256 → H ──────────────→ TSA 签时间 ───────────→ 验证 H == SHA-256(文件) ✅

你的攻击：
文件 → hook → Hₐ ───────────────→ TSA 签时间 ───────────→ 验证 Hₐ ≠ SHA-256(文件) ❌
                                                                    ↑
                                                         验证者自己算哈希，不信任你
```

### 这个攻击到底能干什么？

| 场景 | 结果 |
|---|---|
| 在自己的机器上自欺欺人 | ✅ 你能让本地程序显示"验证通过" |
| 欺骗第三方验证者（法院、审计、用户） | ❌ 绝对失败 |
| 把别人的时间戳令牌贴到你的文件上 | ❌ 验证时哈希不匹配，立刻暴露 |

## 4. 常见疑虑三：重放攻击

### 疑虑

> "截获了一个有效的时间戳令牌，我可以重复使用吗？"

### 解答

可以重复提交同一个令牌给验证者，验证也会通过——但前提是你**同时提交了原始文件**。而验证步骤要求：

```
1. 计算提交文件的哈希
2. 和令牌里的哈希比较
```

如果你提交的文件不是原始文件，哈希对不上，验证失败。

如果你提交了原始文件，那令牌确实有效——**这正是时间戳应该有的行为**。时间戳证明的是"这个文件在这个时间点就已存在"，这个证明对同一文件可以被无限验证。

## 5. 真实攻击面：真正的威胁在哪里

下面这些才是时间戳系统真正需要防范的攻击，而不是哈希碰撞或本地 hook。

### 威胁一：TSA 私钥泄露

| 风险 | TSA 私钥被窃取后，攻击者可伪造任意时间的时间戳 |
|---|---|
| 缓解 | TSA 私钥存放在 HSM（硬件安全模块）中，物理防提取 |
| | TSA 操作有完整审计日志 |
| | 证书吊销机制（CRL/OCSP）可在泄露后废除旧密钥 |

### 威胁二：TSA 作恶（恶意或失职）

| 风险 | TSA 故意在时间戳里写入错误的时间 |
|---|---|
| 缓解 | 选择受 CA/B Forum 审计的公共 TSA |
| | 可同时对多个 TSA 打时间戳，交叉验证 |
| | 时间戳令牌中有 nonce 可防止重放 |

### 威胁三：中间人篡改

| 风险 | 网络层面篡改请求或响应 |
|---|---|
| 缓解 | 使用 HTTPS 传输 TimeStampReq/TimeStampResp |
| | 令牌本身有 TSA 签名，任何篡改都会被检测 |

### 威胁四：签名者密钥泄露（这与 TSA 无关）

| 风险 | 签名用的私钥泄露后，别人可替你的文件签名 |
|---|---|
| 说明 | 这是签名密钥管理的问题，不是时间戳的问题 |
| 缓解 | 使用 HSM 或密钥管理系统保护签名密钥 |
| | 时间戳可以证明——即使密钥泄露，也能分清哪些签名是在泄露前做的 |

## 6. 总结：时间戳的安全边界

### 一句话

> 时间戳的安全不依赖"你的机器可信"，而是依赖**三个独立角色的交叉验证**。

```
        你的机器（计算哈希） + TSA（签发时间戳） + 验证者（独立验证）
             ↓                    ↓                    ↓
          可被欺骗           受 HSM 保护          不可信你的机器
          但没关系              可审计              自己算哈希
```

### 时间戳能保证什么

| 保证 | 说明 |
|---|---|
| 🟢 数据完整性 | 令牌封存的哈希与原始文件绑定，任何篡改可检测 |
| 🟢 时间证据 | TSA 签名证明了时间戳的时间 |
| 🟢 公开可验证 | 任何人拿着文件 + 令牌可独立验证 |
| 🟢 长期有效 | 即使签名证书过期，时间戳仍在有效期内 |

### 时间戳不保证什么

| 不保证 | 说明 |
|---|---|
| 🔴 你的机器是安全的 | 它假定你计算哈希的环境是可信的（这由你负责） |
| 🔴 文件内容是合法的 | 它只证明存在，不证明内容合规 |
| 🔴 TSA 是诚实的 | 这由你选择的 TSA 的信誉保证 |

### 最终结论

```
本地 hook 攻击 → 只能骗自己，骗不了第三方验证者
哈希碰撞      → SHA-256 下当前算力不可行
TSA 作恶      → 选择可信 TSA + 多 TSA 交叉验证
私钥泄露      → HSM 防护 + CRL 吊销
```

时间戳系统的设计是**防御性的**——它假设网络中任何单一角色都可能被攻破，但三个独立角色同时被攻破的概率极低。这就是它的安全本质。
